# 电力设备智能巡检系统开发文档

## 项目目标

本项目在两周内完成一个最小可演示的电力设备智能巡检系统，打通从摄像头采集、Atlas 边缘推理、目标检测、后端存储、告警生成、大模型维修建议到前端展示的完整链路。

开发原则：

- 算法边界清晰：通用目标检测、业务规则和大模型维修建议分开实现。
- 工程归属明确：每个模块都有清晰输入、输出、交付物和验收标准。
- 两周内可完成：优先完成最小完整链路，再补充准确率、稳定性和展示效果。

全组开发必须遵守：

- [数据契约与接口规范](./contracts.md)
- [工程规范与联调标准](./engineering-standards.md)

## 系统总体流程

```text
摄像头采集
   ↓
Atlas 边缘端读取视频
   ↓
目标检测模型
   ├─ 设备识别
   ├─ 故障检测
   └─ 环境异常检测
   ↓
后端接收并保存结果
   ↓
生成告警、巡检记录和大模型维修建议
   ↓
前端可视化展示
```

## 参考实现路线

目标检测和 Atlas 部署优先参考 `atlas-insulator-detection` 的工程路线：

参考仓库：<https://gitee.com/jerry12345555555/atlas-insulator-detection>

```text
数据标注
   ↓
Ultralytics YOLO 训练，得到 best.pt
   ↓
导出 ONNX
   ↓
在昇腾环境用 ATC 转为 OM
   ↓
Atlas 开发板使用 ACL 加载 OM 推理
   ↓
输出检测框、类别、置信度、推理耗时和标注图
   ↓
按 EdgeEye 的 /api/detection/results 契约上传后端
```

本项目借鉴它的训练、转换、板端推理和联调流程，不直接照搬其后端数据契约。EdgeEye 仍以本文档中的 `Detection`、`Fault`、`Alarm`、`Advice` 结构为准。

## 模块划分

| 模块 | 负责人 | 核心问题 |

| Atlas 边缘推理 | 成员1 | 模型如何在开发板上稳定运行 |

## 推荐最终功能范围

必做：

- 3 种左右电力设备识别；
- 2 种左右外观异常检测；
- 1 种环境异常检测；
- Atlas 实际运行模型；
- 后端保存巡检结果；
- 大模型维修建议；
- 前端展示实时和历史结果；
- 生成巡检报告。

选做：

- 将仪表作为普通设备目标进行检测；
- 多种设备类别扩展；
- 更复杂的告警去重和报告分析。

明确不做：

- 仪表读数识别；
- 仪表超限判断；
- 多摄像头；
- 温湿度传感器；
- 复杂用户权限系统。
