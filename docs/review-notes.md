- 模块职责是否清晰；
- 前端页面范围是否和最新要求一致；

## 已定版内容

### 前端页面范围

前端只保留 4 类页面：

- Dashboard；
- 实时巡检；
- 故障中心；
- 报告中心。

页面合并规则：

- 设备识别结果、设备状态、检测框和实时故障放在实时巡检页面；
- 故障、告警、风险等级、处理状态和大模型维修建议放在故障中心页面；
- 摄像头、Atlas、模型和后端状态放在 Dashboard；
- 不再单独设置设备管理页面、仪表记录页面、维修建议页面、系统状态页面。

### 模块职责

- 成员1负责 Atlas、摄像头、边缘推理和上传；
- 成员2负责数据集、目标检测、设备识别和故障检测；
- 成员3负责故障规则、告警规则和大模型维修建议内容设计；
- 成员4负责后端、数据库、报告和大模型调用；
- 成员5负责前端展示、系统测试和答辩材料。

## 关键设计决策

### 仪表方案

当前版本不做仪表识别。

原因：

- 两周内优先保证目标检测、告警、报告和大模型维修建议闭环；
- 仪表读数需要 OCR、阈值、失败兜底和额外测试，会明显增加联调风险；
- 仪表可以作为普通设备类别展示，但不输出读数、单位、读数状态或超限告警。

### 环境异常

环境异常只做一种摄像头可见异常，优先选择 `smoke` 或 `fire`。

原因：

- 可以纳入成员2目标检测模型；
- 不额外引入复杂传感器；
- 前端和后端按普通故障/告警链路处理。

### 明确排除范围

当前版本不做以下能力：

- 多摄像头；
- 温湿度传感器；
- 复杂用户权限系统。

这些能力不进入两周内最小可演示系统，不作为成员分工、接口设计和验收范围。

### 大模型维修建议

大模型维修建议采用“规则先行，大模型增强”的方式：

- 成员3提供规则、提示词和输出结构；
- 成员4负责后端调用、密钥管理、失败重试和结果保存；
- 大模型失败时前端仍可展示规则模板建议，保证演示不中断。
大模型维修建议属于当前必做范围。

### 视频流上传

边缘端采用“连续采集和推理、只上传关键帧”的方式：

- 成员1负责抽帧、相似帧过滤、本地 outbox 队列和幂等上传；
- 正常状态按周期上传 `periodic_sample`；
- 故障首次出现、持续更新、恢复和系统状态变化时上传事件帧；
- 后端按 `idempotencyKey` 防重复保存，按 `eventKey` 聚合同一故障事件；
- 前端展示后端聚合后的 `EventItem`，不自行从原始检测结果派生告警。

### API 响应

后端所有接口统一使用：

```json
{
  "success": true,
  "data": {},
  "message": "ok",
  "timestamp": "2026-06-16T10:00:00+08:00"
}
```

错误统一使用：

```json
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "invalid request payload",
    "details": {}
  },
  "timestamp": "2026-06-16T10:00:00+08:00"
}
```

## 变更控制

任何成员如果要修改以下内容，必须同步更新 `docs/contracts.md`：

- API 路径；
- JSON 字段名；
- 枚举值；
- 错误码；
- 检测结果结构；
- 大模型维修建议结构；
- 前端页面数据需求。
- 关键帧上传字段；
- 事件聚合字段；
- 幂等上传和 ACK 字段。

涉及 API 路径、请求体、响应体、枚举和错误响应的变更，还必须同步更新 `docs/openapi.yaml`。

任何成员如果要修改以下内容，必须同步更新 `docs/engineering-standards.md`：

- 目录结构；
- 文件命名；
- 模型交付格式；
- 规则文件格式；
- 配置项；
- 日志格式；
- 联调准入条件；
- 最终验收条件。
